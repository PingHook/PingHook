Grafana Integration May 2025 · 4 min read

Route Grafana Alerts to Telegram — and stop alert fatigue

Get instant Telegram notifications when Grafana fires an alert — and filter out the noise with one URL parameter. No bot setup, no token juggling — just paste a URL into Grafana's webhook contact point.

The problem with Grafana's native Telegram integration

Grafana has a built-in Telegram contact point. But to use it you need to:

  1. Create a Telegram bot via @BotFather and generate a token
  2. Find your Telegram chat ID (send a message and hit the API to retrieve it)
  3. Paste both into Grafana
  4. Repeat for every team member who wants alerts

It works, but it's four steps of friction before you see a single notification. And if you want to fan out to Slack or Discord as well, you're setting up separate integrations for each platform.

PingHook reduces this to one step. You get a webhook URL from the bot. Paste it into Grafana. Done.

Prerequisites

  • Grafana with unified alerting (version 8.x or later)
  • A Telegram account
  • About 30 seconds

Step 1 — Get your PingHook URL

Open Telegram and message @PingHookBot. Send /start. The bot replies with your unique webhook URL instantly — no email, no password, no signup.

https://pinghook.dev/send/YOUR_KEY

Step 2 — Add a Webhook contact point in Grafana

  1. Go to Alerting → Contact points → New contact point
  2. Set the type to Webhook
  3. Paste your URL with a label and a filter:
https://pinghook.dev/send/YOUR_KEY/grafana?if=status:eq:firing

The ?if=status:eq:firing filter is the critical part. Grafana sends a webhook for both firing and resolved events — without this, you get notified every time an alert recovers too. The filter keeps only what's actionable.

  1. Click Test — Grafana sends a sample payload and you'll see a message in Telegram within seconds
  2. Save the contact point and assign it to a notification policy

What the notification looks like

Grafana sends a structured JSON payload. PingHook auto-formats it as a readable code block in Telegram:

# You receive this in Telegram: 🔔 [grafana] { "status": "firing", "receiver": "pinghook", "commonLabels": { "alertname": "HighCPU", "env": "production" }, "commonAnnotations": { "summary": "CPU > 90% for 5 minutes" } }
You can slim down the payload in Grafana's contact point settings — include only the fields you care about to keep the notification short.

Advanced: filtering and routing

Only prod alerts

Chain multiple ?if= params — all must pass (AND logic):

?if=status:eq:firing&if=commonLabels.env:eq:production

Suppress alert storms

Add ?dedup=10 to suppress the same alert for 10 minutes — so a flapping service doesn't flood your chat:

?if=status:eq:firing&dedup=10

Route critical alerts to Telegram, others to Slack

Create two contact points in Grafana — each pointing to a different PingHook URL with a ?channel= param:

# Contact point 1 — critical → Telegram only ?if=status:eq:firing&if=commonLabels.severity:eq:critical&channel=telegram # Contact point 2 — everything else → Slack only ?if=status:eq:firing&channel=slack

Both URLs share the same API key. The ?channel= param handles the routing — your other channels stay connected and unaffected.

Fan out to Slack and Discord at once

Connect additional channels in the Telegram bot — then every Grafana alert lands everywhere simultaneously:

/connect slack https://hooks.slack.com/services/... /connect discord https://discord.com/api/webhooks/...

Troubleshooting

Test ping works, real alerts don't appear. Check your notification policy — the contact point needs to be in the routing tree. Run /history in the bot to see if pings are arriving but being suppressed.

Still getting "resolved" notifications. Confirm ?if=status:eq:firing is in the URL. Without it, PingHook delivers both firing and resolved events.

Alert storm flooding your chat. Add ?dedup=N (minutes). The first alert comes through; subsequent ones with the same label are suppressed until the window expires.

Want to inspect what Grafana is actually sending. Use ?silent=1 temporarily — pings are logged but not delivered. Then check /history in the bot to see the raw payload.
